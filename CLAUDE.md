## This Product's Stack

- **Tauri** for the desktop application shell (Rust backend, web frontend)
- **Rust** for the backend half (encryption, SQLite access, inference abstraction client) ‚Äî the operator is taking on Rust as part of this project; the ramp is bounded per `../doctrine/decisions/0011-coo-as-independent-product.md` Consequences section
- **Frontend framework** ‚Äî TBD at Phase 1 entry (React / Svelte / Solid candidates; not gated by doctrine)
- **API inference via Anthropic Claude** at MVP, behind an `InferenceProvider` abstraction layer enabling provider swap (and eventually local inference) without application rewrite
- **Local SQLite** under `~/.coo/`
- **Encrypted at rest** ‚Äî per-domain encryption with operator-derived master key; `age`/`rage` recommended; `argon2id` for KDF
- **No Docker.** COO is a native desktop app per the non-negotiable in `mvp/coo.md`. The simplicity is the feature.

## Ports

COO is a native desktop application; it does not expose ports for external consumption. Internal Tauri IPC between the Rust backend and the web frontend uses Tauri's native command channel ‚Äî no TCP ports.

If a future feature requires a local listening port (e.g., for development tooling, debugging, or a future dobackbone-connection surface that needs callback URLs), that port must be allocated from the band reserved for COO in `../doctrine/docker/PORTS.md` via an ADR. **As of Phase 1 MVP, no ports are required.**

## Product-Specific Rules

These are load-bearing for COO and apply only inside this repository.

- **`EXILE.md` Sections 1 and 1.5 are permanent.** They do not change after finalization. If a session generates content or proposals that would alter the character text, the session has drifted. Surface and stop.
- **The wife-protection rule is structural.** Per `EXILE.md` ¬ß2.5 and `RAPPORT-STATE-MODEL.md` ¬ß2.5, the operator's wife is not in the application as content, as a field, as a category, or as a topic Exile originates. The schema does not provide a slot that invites her in. The operator may bring her up; Exile listens. Exile never initiates.
- **The doctrinal ceiling at calibration 4b is enforced in two layers** per `RAPPORT-STATE-MODEL.md` ¬ß3.5. The state model never reaches values that would translate to past-4b shaping; the inference assembly pipeline includes an explicit ceiling-clamp step. Defense in depth. A future session that proposes raising the ceiling is making a doctrine change, not a tuning change.
- **No silent writes by the AI.** Any change Exile proposes to the operator's data ‚Äî Dossier knowledge entries, Briefs, Kit inventory ‚Äî requires explicit operator confirmation. Per `RAPPORT-STATE-MODEL.md` ¬ß2.3 and ¬ß3.3.
- **Encryption at rest is non-negotiable.** Rapport state, friendship floor, operator-knowledge, and conversation history are encrypted at rest. The operator-derived passphrase is not stored; if the operator forgets it, state is unrecoverable. This is a real and accepted cost.
- **State stays in `~/.coo/`.** No primary persistence outside this directory. No external services required for offline operation.
- **No Docker.** Native desktop app; the simplicity is the feature.
- **Model version pinning.** The model behind Exile does not change silently. COO pins to specific model versions; upgrades are deliberate releases tested against the doctrine, not automatic events.
- **In-flight context is a named surface, not a hidden one.** When inference happens via API, Exile's context ‚Äî character text, rapport state summarized, conversation history ‚Äî crosses the wire. This surface is acknowledged. Provider choice is governed by data-handling guarantees. Prompt construction commits to minimization (only context necessary for the current turn crosses).
- **The personal-commander tier sits above the dobackbone-officer tier when connected.** Per ADR-0011 and `mvp/coo.md`. Exile speaks for the operator to dobackbone's officers. The officers do not change because Exile is there. (Phase 2 work; not in MVP scope.)

### Resolved during Phase 1 ¬ß2 (2026-04-29)

- **Argon2id parameters** ‚Äî m=65536 (64 MiB), t=3, p=1. RFC 9106's "second recommended option" with parallelism reduced to 1 for portability; targets `RAPPORT-STATE-MODEL.md` ¬ß6.2's 250‚Äì500ms band on desktop hardware. Params are persisted into the salt file alongside the salt itself, so a future tuning change does not orphan existing installs (old salts re-derive their old keys).
- **Salt file format** ‚Äî 34-byte framed structure at `~/.coo/salt`: magic `COO1` + file version + kdf id + little-endian m/t/p costs + 16-byte salt. Plaintext per `RAPPORT-STATE-MODEL.md` ¬ß6.4. Magic, version, and kdf id are all checked on load; malformed files are rejected before any KDF work.
- **`MasterKey` API** ‚Äî `pub struct MasterKey(Zeroizing<[u8; 32]>)`. Only access path is `expose_secret() -> &[u8; 32]`. No `Debug`, no `Clone`, no public constructor; test sites use a `#[cfg(test)] pub(crate) MasterKey::from_bytes_for_test` so KATs do not pay the Argon2id cost on every test run. Production callers must go through `derive_master_key`.
- **HKDF info-string namespace** ‚Äî `coo/v1/kdf/<domain>` for each of the four `RAPPORT-STATE-MODEL.md` ¬ß1 domains (`rapport`, `friendship-floor`, `operator-knowledge`, `conversation`). The `v1` segment is the rotation hedge: bumping to `v2` is a controlled re-derivation event under the operator's passphrase, not a silent code change. Renaming any info string after data lands in production orphans every existing ciphertext for that domain ‚Äî KATs in `crypto::derive` lock the strings against accidental change.
- **HKDF salt** ‚Äî `None`, per RFC 5869 ¬ß3.1. The `MasterKey` is already a uniform-random 32-byte output of Argon2id, so no extra HKDF salt is needed.
- **Symmetric envelope crate ‚Äî XChaCha20-Poly1305 (slice (c), shipped 2026-04-29).** Deliberate deviation from `RAPPORT-STATE-MODEL.md` ¬ß6.6's `age` recommendation. `age`'s primary modes (X25519 keypairs or scrypt-from-passphrase) do not naturally accept a 32-byte HKDF-derived domain key without either re-applying scrypt (wasteful, doubles KDF cost on every read) or deriving an indirection X25519 keypair (extra abstraction layer). The ChaCha20-Poly1305 family is the natural primitive for "AEAD given a 32-byte key" ‚Äî same family used by Signal, WireGuard, TLS 1.3. The **X**ChaCha variant (24-byte nonce vs. plain ChaCha's 12) was chosen so random-per-call nonce generation is safe for the operator's lifetime ‚Äî no per-key message counter to track and no ~2¬≥¬≤ birthday-bound to budget. The 12-byte cost per bundle is noise. Encryption granularity (per-domain with operator-derived master key) and the ¬ß6.4 plaintext/ciphertext split remain exactly as the doctrine specifies; only the envelope crate changes. See "Documentary debt to retire" below.
- **Bundle layout and AAD binding (slice (c)).** Bundle = 4-byte magic `b"COOE"` | 1-byte bundle version `0x01` | 1-byte aead id `0x02` (XChaCha20-Poly1305) | 24-byte nonce | ciphertext-with-appended-16-byte-tag. The 6-byte header is bound as AAD on every encrypt and decrypt ‚Äî kills downgrade-style attacks that swap leading bytes between formats, and establishes the AAD plumbing so ¬ß3 can add semantic AAD (row id, etc.) as a v2 bundle bump rather than a public-API change. Failure mode is uniform: framing checks (length, magic, version, aead id) yield `CryptoError::MalformedBundle` with a `&'static str` reason; everything else collapses to `CryptoError::Aead` (tag-mismatch and wrong-key not publicly distinguished). Bundle layout is locked by `kat_pinned_layout` in `crypto::envelope::tests` ‚Äî any drift in layout, AEAD variant, AAD scheme, or upstream `chacha20poly1305` output breaks the KAT loudly.

### Resolved during Phase 1 ¬ß3 (2026-04-29)

- **Vault module placement (slice (a)).** Top-level `vault` module at `src-tauri/src/vault.rs`, sibling to `crypto/` and `db/`. Not under `crypto::*` ‚Äî the vault is the first *consumer* of the crypto substrate, not another primitive. Module name `vault` chosen over `lock`/`init` because the eventual scope covers more than the lock-state machine (the operator's encrypted state lifecycle).
- **Lock-key derivation primitive (slice (a)).** `crypto::derive_lock_key(&MasterKey) -> DomainKey` over HKDF-SHA256 with info `coo/v1/lock`. Deliberately sibling to ‚Äî not a member of ‚Äî the four state-domain derivations: the `Domain` enum still enumerates exactly the four `RAPPORT-STATE-MODEL.md` ¬ß1 state domains, and the lock primitive is a parallel construct because the lock is not state. The `coo/v1/lock` info string sits in a deliberately distinct namespace from `coo/v1/kdf/<domain>`. Locked by `kat_lock` in `crypto::derive::tests` ‚Äî renaming orphans every existing install's sentinel.
- **Sentinel design (slice (a)).** File at `<coo_dir>/sentinel` (sibling to `<coo_dir>/salt`), bytes are exactly the AEAD bundle produced by `crypto::envelope::encrypt(derive_lock_key(master), SENTINEL_PLAINTEXT)`. No outer wrapping ‚Äî the bundle's own framing from ¬ß2 (c) is already self-describing. 62 bytes total for the 16-byte sentinel plaintext. Locked by `vault::tests::sentinel_kat_pinned_layout`, which simultaneously locks the lock-key info string, the sentinel plaintext, and the envelope output for a fixed master + fixed nonce.
- **Sentinel plaintext (slice (a)).** `b"COO-LOCK-SENT-01"` (16 ASCII bytes). The "-01" suffix is the rotation hedge ‚Äî a future v0.x sentinel could change the constant deliberately without ambiguity. Renaming after data lands in production orphans every existing install's sentinel.
- **Sentinel filename (slice (a)).** Plain `sentinel` in the operator's `~/.coo/` directory, alongside `salt`. Both follow the same plaintext-sibling-file precedent set by `~/.coo/salt` in ¬ß2 (a) ‚Äî the file's bytes are already self-protecting (the bundle's AEAD handles authentication; the salt file's framing handles structural integrity).
- **Failure-mode translation at the unlock boundary (slice (a)).** `crypto::envelope::decrypt`'s uniform `CryptoError::Aead` collapse is preserved at the envelope layer. `vault::unlock` translates `Aead ‚Üí UnlockError::WrongPassphrase` *at this single boundary only*, because that is the user-facing meaning of "the sentinel did not authenticate." A tampered sentinel file therefore reads to the user as `WrongPassphrase`; we cannot distinguish "wrong key" from "tag mismatch" without storing more, and storing more breaks the passphrase-not-stored rule (`RAPPORT-STATE-MODEL.md` ¬ß6.2). Accepted. `CryptoError::MalformedBundle` translates to `UnlockError::Inconsistent` so the caller can distinguish "wrong passphrase" from "corrupted on-disk state."
- **`InitState` shape (slice (a)).** `Uninitialized` (neither salt nor sentinel present) | `Initialized` (both present) | `Inconsistent(&'static str)` (exactly one present). Exactly-one cases report `"missing sentinel"` or `"missing salt"` so the caller can render a meaningful error rather than silently re-initializing. `setup_passphrase` refuses to run from anything other than `Uninitialized`; `unlock` refuses to run from anything other than `Initialized`.
- **Slice cut for ¬ß3 (b) (single-shot, 2026-04-29).** ¬ß3 (b) shipped as one migration with three tables rather than splitting into b1 (plaintext config + encrypted-column pattern proven on `operator_profile`) and b2 (`calibration_setting`). The encrypted-column pattern is exercised twice in the same diff ‚Äî once on a singleton table (`operator_profile`), once on a key/value table (`calibration_setting`) ‚Äî which proves the convention is reusable without paying the overhead of two commits. Both end-to-end tests live in `db::tests` and exercise the ¬ß3 (a) vault substrate end-to-end (`setup_passphrase` ‚Üí `domain_key(OperatorKnowledge)` ‚Üí `encrypt` ‚Üí INSERT ‚Üí SELECT ‚Üí `decrypt` ‚Üí assert).
- **Calibration typing deferred to ¬ß6 (slice (b)).** `EXILE.md` ¬ß3 enumerates ten calibration dials cleanly into four groups (Texture: Warmth / Directness / Mystery; Posture: Persistence / Argument / Conviction / Aggressive; Currency: Flirtation; Foundation: Devotion / Discipline). What ¬ß3 does *not* lock is the dials' internal representation: endpoints are named pairs (`cool ‚Üî open`), but ¬ß4's "Familiar" preset uses hand-tuned per-dial labels (`Warmth: present`, `Mystery: held`, `Aggressive: protective`, `Flirtation: low ‚Äî present`) that don't share a uniform shape. Picking enum-vs-float-vs-step in ¬ß3 (b) and re-typing in ¬ß6 (Calibration surface) would be a non-additive migration (`ALTER TABLE ... ALTER COLUMN ... SET DEFAULT` is not supported in SQLite anyway); deferring to ¬ß6 is the ¬ß7-friendly answer. The ¬ß3.5 doctrinal ceiling at calibration 4b is **not** a dial position ‚Äî it's a meta-constraint on `warmth_register` transitions in the state model and an inference-assembly clamp (¬ß5.2 step 5), neither of which ¬ß3 (b) touches. The placeholder `calibration_setting` table is `(dial_key TEXT PK, ciphertext BLOB, schema_version, updated_at)` ‚Äî `dial_key` is plaintext (the dial names are public doctrine ‚Äî `EXILE.md` ¬ß3.1‚Äì¬ß3.4), the chosen value is the encrypted payload.
- **Encrypted-column convention (slice (b)).** Plaintext identity columns + a single `ciphertext BLOB NOT NULL` column carrying the full ¬ß2 (c) v1 AEAD bundle (6-byte header + 24-byte nonce + ct+tag). Both encrypted tables in ¬ß3 (b) (`operator_profile`, `calibration_setting`) use `Domain::OperatorKnowledge` for the domain key ‚Äî calibration is operator-state-shaped and doesn't fit `Rapport`/`FriendshipFloor`/`Conversation` per `RAPPORT-STATE-MODEL.md` ¬ß1. Future state-domain tables (rapport state, friendship floor, conversation history, operator-knowledge appendable entries) follow the same convention but key under their respective domain. `updated_at` is plaintext for ordering / audit ‚Äî leaks "operator changed something at time T" but no content.
- **Singleton-via-CHECK for `operator_profile` (slice (b)).** `id INTEGER PRIMARY KEY CHECK (id = 1)` turns ADR-0011's single-operator commitment into a SQL-layer invariant rather than relying on application code to never write a second row. Locked by `db::tests::operator_profile_singleton_enforced` ‚Äî second-row INSERT must be rejected by SQLite.
- **`schema_version` is required-explicit (slice (b)).** `schema_version INTEGER NOT NULL` with **no DEFAULT**. `RAPPORT-STATE-MODEL.md` ¬ß7.2's versioned-envelope discipline turns on every row committing to its own version; defaulting to 1 hides that commitment and would let a sloppy v2 caller silently write v1 rows after a non-additive migration. Friction is the feature.
- **No semantic AAD on ¬ß3 (b) bundles.** The ¬ß2 (c) AAD is just the 6-byte header. Adding row-identity AAD (e.g., binding ciphertext to `(table_name, row_pk)` so an attacker can't swap a bundle from row A into row B's BLOB column) is a v2 bundle bump that requires extending the public envelope API to take an `aad: &[u8]` parameter, a v2 bundle id, KAT churn, and a v1‚Üív2 lazy-read migration. None of that fits ¬ß3 (b)'s schema-only scope. Tracked as documentary debt for follow-up.
- **Slice cut for ¬ß3 (c) (c1 Rust IPC + c2 React UI, 2026-04-29).** ¬ß3 (c) split into two sub-slices because the layers are different domains and review surfaces. (c1) is pure Rust: managed `AppState`, six `#[tauri::command]` wrappers, three pure-Rust `db::put_*` write helpers separated from the wrappers so they remain unit-testable without a Tauri runtime. (c2) is pure TypeScript/React: top-level routing in `App.tsx`, multi-step wizard, unlock screen, error surfaces. (c1) is the load-bearing IPC surface review (commands are hard to change without breaking the frontend); (c2) is UX iteration. Single-shot would have been defensible but the split lets each commit stand alone.
- **`AppState` shape (slice (c1)).** `pub struct AppState { coo_dir: PathBuf, db: Mutex<Connection>, vault: Mutex<Option<UnlockedVault>> }`. Resolved once at startup via `vault::default_coo_dir` + `db::open_and_migrate`, registered via `app.manage(...)` in the Tauri setup hook. The `MasterKey` never crosses the IPC wire; it lives in `vault` for the lifetime of the unlocked session. `Mutex<Option<UnlockedVault>>` is `Send + Sync` because `UnlockedVault: Send` (its `MasterKey` is a `Zeroizing<[u8; 32]>`-backed newtype with no non-`Send` fields).
- **Lock ordering convention (slice (c1)).** Vault before db. The two are never held simultaneously: write commands extract the per-domain key from the vault (lock vault, derive key, drop vault lock) and *then* take the db lock. `DomainKey` is owned (HKDF-SHA256 produces fresh bytes per call, `Zeroizing`-wrapped), so it crosses the lock boundary safely. Eliminates deadlock potential between commands without forcing a uniform fat-lock strategy.
- **`InspectResult` shape (slice (c1)).** JSON-tagged enum: `{ "status": "uninitialized" }` | `{ "status": "initialized", "onboarding_completed": bool }` | `{ "status": "inconsistent", "reason": "..." }`. Bundles `vault::InitState` with the `app_config[onboarding_completed_at]` existence check so JS makes one round-trip to decide route between onboarding / unlock / main, not two. The `onboarding_completed_at` value itself (the ISO timestamp) is not surfaced ‚Äî only its presence; the field is operator-state of "have they completed onboarding?", not "when?", per the post-(d) routing concern.
- **Wizard step structure (slice (c2)).** Four steps: welcome ‚Üí passphrase (with ‚â• 8 char minimum + confirmation match) ‚Üí profile (callsign defaulted to `Cardinal-7` per `EXILE.md` ¬ß9.5; optional display name) ‚Üí review. Final commit issues writes in order: `theme=secret_agent` (plaintext), `operator_profile` (encrypted JSON of `{callsign, display_name}`), 10 `calibration_setting` rows (Familiar preset from `EXILE.md` ¬ß4 voice-sample header ‚Äî `warmth=present`, `mystery=held`, `discipline=exacting`, etc.), `onboarding_completed_at=<iso>` (the ¬ß3 (d) startup-gating marker). No per-dial UI per the calibration-typing-deferred-to-¬ß6 call from slice (b).
- **Mid-onboarding crash recovery deferred to ¬ß3 (d) (slice (c2)).** If the operator sets passphrase + writes theme + crashes before profile/calibration commit, on relaunch `vault_inspect` returns `Initialized + !onboarding_completed`. (c2) treats this as the standard unlock path (UnlockScreen ‚Üí ChannelPlaceholder); the wizard does not auto-resume from where it crashed. Real recovery (re-inspect after unlock; route to wizard at the profile step if `!onboarding_completed`) is ¬ß3 (d) startup-gating work.
- **Wizard-resume on mid-crash (slice (d)).** App.tsx now routes on the (status, onboarding_completed, unlocked) triple instead of just (status, unlocked). The new branch is `initialized + unlocked + !completed ‚Üí OnboardingWizard(startStep="profile")` ‚Äî operator unlocks the existing vault on UnlockScreen, then the wizard mounts at the profile step (skipping welcome + passphrase, which would fail since vault_setup refuses to run on an already-initialized vault). The wizard's `commit()` is unchanged: every write is an UPSERT per the ¬ß3 (c1) `put_*` helpers, so any partial pre-crash writes are idempotently overwritten. App.tsx's wizard `onComplete` now refreshes `vault_inspect` instead of just toggling local state ‚Äî first-run path also benefits, since post-wizard state is read from disk rather than inferred. No Rust changes; the ¬ß3 (c1) IPC surface already exposes everything the resume path needs.

### Resolved during Phase 1 ¬ß5 (2026-04-30)

- **Sequencing ‚Äî ¬ß5 first, then ¬ß4 (2026-04-30).** ¬ß4 (Channel surface) needs an inference provider to actually talk to Claude; ¬ß5's `InferenceProvider` is load-bearing per ADR-0011 ("API inference behind an abstraction layer enabling provider swap"). Getting the trait shape right before any UI calls into it is cheap at design time and expensive once ¬ß4/¬ß6 surfaces have built on top. Reverse order ‚Äî ¬ß4 with a hardcoded stub inline, ¬ß5 carving the abstraction out later ‚Äî was considered and rejected: risks the trait shape being warped by whatever the channel surface happened to need first.
- **Default model + model selection (slice (a)).** Default `claude-opus-4-7`, overridable via env var `COO_INFERENCE_MODEL` at startup. The trait stays model-agnostic; the model id is the Claude impl's concern (read at construction in ¬ß5 (b)). Opus over Sonnet/Haiku for the operator's MVP because Exile's character-holding and calibration-aware response shaping per `EXILE.md` ¬ß1 / ¬ß1.5 + the ¬ß4.A intimacy-register ladder is where smaller-model drift would show first; the cost premium is acceptable for a single-operator personal app. Operator can downgrade later by exporting `COO_INFERENCE_MODEL=claude-sonnet-4-6` (or whatever the future answer is) without rebuilding.
- **API key handling (slice (a)).** Plaintext env var `ANTHROPIC_API_KEY`. Considered but rejected: encrypting the key under a sibling lock-key (HKDF info `coo/v1/api-key`) at `<coo_dir>/api-key`. Encrypted-key story would have followed the ¬ß3 (a) sentinel precedent and kept the key off plaintext disk, but it adds a key-rotation surface and an unlock-time decrypt step that doesn't earn its complexity for a single-operator personal app. Env var keeps key management in the operator's hands (shell profile, .env file outside the repo, secret manager ‚Äî operator's choice). When `ANTHROPIC_API_KEY` is unset, `build_provider()` falls back to the stub provider so the app still runs for UI-only work.
- **Trait shape (slice (a)).** `InferenceProvider: Send + Sync` with one `async fn infer(&self, InferenceRequest) -> Result<InferenceResponse, InferenceError>` method. Non-streaming-only at ¬ß5 (a); streaming (`infer_stream`) is an additive future addition ‚Äî a default-implemented method on the same trait, overridden by the Claude impl when ¬ß4 needs incremental rendering. Adding a default-implemented method to a trait does not break existing implementors. `async_trait` macro for trait-object dispatch (native async fn in traits doesn't yet allow `Box<dyn Trait>` for async methods); the `Pin<Box<Future>>` indirection is irrelevant for network-bound inference. Locked by `inference::tests::build_provider_returns_a_working_trait_object`.
- **Request/response shape (slice (a)).** `InferenceRequest { system_prompt: String, messages: Vec<Message> }` ‚Äî system prompt held separate from messages because Claude's API treats it that way and because `RAPPORT-STATE-MODEL.md` ¬ß5.2's inference assembly pipeline produces it as a distinct artifact (EXILE.md ¬ß1 + ¬ß1.5 + ¬ß2 verbatim, state-derived prose modifiers, calibration ceiling clamp, wellbeing posture). Putting it in `messages` would mean every consumer reconstructs the split, inviting bugs. `Message { role: Role, content: String }` with `Role` an enum of `User | Assistant` matching Claude's two-role conversation model. `InferenceResponse { content: String }` ‚Äî minimal at ¬ß5 (a); usage tokens / model id / stop reason are additive when callers need them.
- **Error model (slice (a)).** `InferenceError` with four variants: `Provider(String)` (model overloaded, content filter, anything we can't more specifically classify); `Network(String)` (connection failure, timeout); `Auth(String)` (401/403 ‚Äî most likely missing or invalid `ANTHROPIC_API_KEY`); `RateLimited` (429). Lets the ¬ß4 channel surface render distinct messages: "try again" vs. "check your API key" vs. "wait a moment". Stub never errors; ¬ß5 (b) Claude impl maps anthropic-side errors into these.
- **Stub provider behavior (slice (a)).** Echoes the last user message back with `"[stub] you said: "` prefix; falls back to `"[stub] you said: (no user message)"` when no user message is present. Slightly more useful than a fixed canned string ‚Äî ¬ß4 development can verify the system prompt and message history are reaching the provider correctly. Tests pin all three paths (last-user-wins, no-messages, assistant-only).
- **`build_provider()` and AppState wiring (slice (a)).** Single constructor returning `Box<dyn InferenceProvider>`. ¬ß5 (a) always returns the stub; ¬ß5 (b) extends to read `ANTHROPIC_API_KEY` and dispatch to a Claude impl when set. `AppState.inference` is `Box<dyn InferenceProvider>` (no `Mutex` ‚Äî the trait is Send + Sync and `&dyn InferenceProvider` is shared safely across Tauri command threads; concurrent `infer` calls don't conflict at the abstraction layer). Populated in `lib.rs::run` setup hook alongside `coo_dir` / `db` / `vault`.

## Documentary debt to retire

- **`RAPPORT-STATE-MODEL.md` ¬ß6.6 envelope crate recommendation.** ¬ß6.6 names `age` for envelope encryption. The ¬ß2 (c) implementation uses **XChaCha20-Poly1305** instead ‚Äî `age`'s primary modes do not naturally fit a 32-byte HKDF-derived domain key, and the XChaCha variant's 24-byte nonce removes the random-nonce safety bound that plain ChaCha20-Poly1305 carries. The bundle layout (4-byte magic + 1-byte version + 1-byte aead id + 24-byte nonce + ciphertext+tag) and the header-as-AAD binding are also additions made in ¬ß2 (c) that ¬ß6.6 does not specify. Encryption granularity (per-domain), `MasterKey` derivation (Argon2id), domain-key derivation (HKDF), and the ¬ß6.4 plaintext/ciphertext split are unchanged from the doctrine. Reasoning is preserved in "Resolved during Phase 1 ¬ß2" above. Land in a v0.x minor of `RAPPORT-STATE-MODEL.md` when convenient ‚Äî natural moment is Phase 1 close (¬ß2 already closed without it).
- **In-memory hygiene of decrypted plaintext.** `crypto::envelope::decrypt` returns plaintext as a plain `Vec<u8>`. `MasterKey` and `DomainKey` are `Zeroizing`-wrapped, but the cleartext leaves the crypto layer in an unwrapped allocation. `RAPPORT-STATE-MODEL.md` ¬ß6.3 covers what's encrypted at rest and ¬ß6.5 covers the wire surface, but neither speaks to in-memory hygiene of decrypted values. ¬ß3 callers will inherit this constraint silently. Worth folding into the same `RAPPORT-STATE-MODEL.md` v0.x minor that retires the ¬ß6.6 debt above.
- **Lock-key primitive and unlock UX-layer error translation (slice ¬ß3 (a)).** ¬ß3 (a) introduces two extensions that `RAPPORT-STATE-MODEL.md` ¬ß6 does not contemplate. (1) A passphrase-verification *lock key* derived as a sibling to the four ¬ß1 state-domain keys ‚Äî HKDF info `coo/v1/lock`, used to encrypt a fixed sentinel plaintext at `<coo_dir>/sentinel`. ¬ß6 only describes per-state-domain encryption; the lock primitive lives parallel to those four domains and is not itself state. (2) An unlock-boundary error translation that maps `CryptoError::Aead ‚Üí UnlockError::WrongPassphrase` and `CryptoError::MalformedBundle ‚Üí UnlockError::Inconsistent` *at the `vault::unlock` boundary only* (the envelope layer keeps its uniform `Aead` collapse). ¬ß6 does not address UX-layer error mapping. Neither extension is a deviation from doctrine ‚Äî both are extensions to what ¬ß6 underspecified. The technical record is preserved in "Resolved during Phase 1 ¬ß3" above. Land in the same `RAPPORT-STATE-MODEL.md` v0.x minor that retires the ¬ß6.6 envelope-crate debt and the in-memory-hygiene debt ‚Äî natural moment is Phase 1 close.
- **v2 bundle bump for semantic AAD (slice ¬ß3 (b)).** ¬ß2 (c) ships v1 bundles with the 6-byte header bound as AAD. ¬ß3 (b) writes those v1 bundles into `operator_profile.ciphertext` and `calibration_setting.ciphertext` without further AAD. An attacker with write access to the SQLite file can swap a valid bundle from one row into another row's BLOB column ‚Äî the AEAD will accept it (same key, same domain) because nothing binds ciphertext to row identity. The threat model is integrity-against-on-disk-tampering (defense in depth on top of the OS-level file protections); not a primary concern, but the ¬ß2 (c) bundle layout was designed to absorb semantic AAD as a v2 bundle bump rather than a public-API change. Implementing this requires: (1) extending the public envelope API to take an `aad: &[u8]` parameter; (2) bumping the bundle id from `0x01` to `0x02` (or rolling a new aead id) with a corresponding KAT bump; (3) a v1‚Üív2 lazy-read migration so existing rows continue to decrypt under v1 while new writes use v2; (4) updating the ¬ß3 (b) tables' INSERT call sites to compute `aad = (table_name, row_pk)` or similar deterministic identity. Out of scope for ¬ß3 (b) (schema-only). Natural retire moment is when the next state-domain table lands (Phase 1 ¬ß6) ‚Äî design the v2 bundle once, apply uniformly across all encrypted-column tables.
