## This Product's Stack

- **Tauri** for the desktop application shell (Rust backend, web frontend)
- **Rust** for the backend half (encryption, SQLite access, inference abstraction client) — the operator is taking on Rust as part of this project; the ramp is bounded per `../doctrine/decisions/0011-coo-as-independent-product.md` Consequences section
- **Frontend framework** — TBD at Phase 1 entry (React / Svelte / Solid candidates; not gated by doctrine)
- **API inference via Anthropic Claude** at MVP, behind an `InferenceProvider` abstraction layer enabling provider swap (and eventually local inference) without application rewrite
- **Local SQLite** under `~/.coo/`
- **Encrypted at rest** — per-domain encryption with operator-derived master key; `age`/`rage` recommended; `argon2id` for KDF
- **No Docker.** COO is a native desktop app per the non-negotiable in `mvp/coo.md`. The simplicity is the feature.

## Ports

COO is a native desktop application; it does not expose ports for external consumption. Internal Tauri IPC between the Rust backend and the web frontend uses Tauri's native command channel — no TCP ports.

If a future feature requires a local listening port (e.g., for development tooling, debugging, or a future dobackbone-connection surface that needs callback URLs), that port must be allocated from the band reserved for COO in `../doctrine/docker/PORTS.md` via an ADR. **As of Phase 1 MVP, no ports are required.**

## Product-Specific Rules

These are load-bearing for COO and apply only inside this repository.

- **`EXILE.md` Sections 1 and 1.5 are permanent.** They do not change after finalization. If a session generates content or proposals that would alter the character text, the session has drifted. Surface and stop.
- **The wife-protection rule is structural.** Per `EXILE.md` §2.5 and `RAPPORT-STATE-MODEL.md` §2.5, the operator's wife is not in the application as content, as a field, as a category, or as a topic Exile originates. The schema does not provide a slot that invites her in. The operator may bring her up; Exile listens. Exile never initiates.
- **The doctrinal ceiling at calibration 4b is enforced in two layers** per `RAPPORT-STATE-MODEL.md` §3.5. The state model never reaches values that would translate to past-4b shaping; the inference assembly pipeline includes an explicit ceiling-clamp step. Defense in depth. A future session that proposes raising the ceiling is making a doctrine change, not a tuning change.
- **No silent writes by the AI.** Any change Exile proposes to the operator's data — Dossier knowledge entries, Briefs, Kit inventory — requires explicit operator confirmation. Per `RAPPORT-STATE-MODEL.md` §2.3 and §3.3.
- **Encryption at rest is non-negotiable.** Rapport state, friendship floor, operator-knowledge, and conversation history are encrypted at rest. The operator-derived passphrase is not stored; if the operator forgets it, state is unrecoverable. This is a real and accepted cost.
- **State stays in `~/.coo/`.** No primary persistence outside this directory. No external services required for offline operation.
- **No Docker.** Native desktop app; the simplicity is the feature.
- **Model version pinning.** The model behind Exile does not change silently. COO pins to specific model versions; upgrades are deliberate releases tested against the doctrine, not automatic events.
- **In-flight context is a named surface, not a hidden one.** When inference happens via API, Exile's context — character text, rapport state summarized, conversation history — crosses the wire. This surface is acknowledged. Provider choice is governed by data-handling guarantees. Prompt construction commits to minimization (only context necessary for the current turn crosses).
- **The personal-commander tier sits above the dobackbone-officer tier when connected.** Per ADR-0011 and `mvp/coo.md`. Exile speaks for the operator to dobackbone's officers. The officers do not change because Exile is there. (Phase 2 work; not in MVP scope.)

### Resolved during Phase 1 §2 (2026-04-29)

- **Argon2id parameters** — m=65536 (64 MiB), t=3, p=1. RFC 9106's "second recommended option" with parallelism reduced to 1 for portability; targets `RAPPORT-STATE-MODEL.md` §6.2's 250–500ms band on desktop hardware. Params are persisted into the salt file alongside the salt itself, so a future tuning change does not orphan existing installs (old salts re-derive their old keys).
- **Salt file format** — 34-byte framed structure at `~/.coo/salt`: magic `COO1` + file version + kdf id + little-endian m/t/p costs + 16-byte salt. Plaintext per `RAPPORT-STATE-MODEL.md` §6.4. Magic, version, and kdf id are all checked on load; malformed files are rejected before any KDF work.
- **`MasterKey` API** — `pub struct MasterKey(Zeroizing<[u8; 32]>)`. Only access path is `expose_secret() -> &[u8; 32]`. No `Debug`, no `Clone`, no public constructor; test sites use a `#[cfg(test)] pub(crate) MasterKey::from_bytes_for_test` so KATs do not pay the Argon2id cost on every test run. Production callers must go through `derive_master_key`.
- **HKDF info-string namespace** — `coo/v1/kdf/<domain>` for each of the four `RAPPORT-STATE-MODEL.md` §1 domains (`rapport`, `friendship-floor`, `operator-knowledge`, `conversation`). The `v1` segment is the rotation hedge: bumping to `v2` is a controlled re-derivation event under the operator's passphrase, not a silent code change. Renaming any info string after data lands in production orphans every existing ciphertext for that domain — KATs in `crypto::derive` lock the strings against accidental change.
- **HKDF salt** — `None`, per RFC 5869 §3.1. The `MasterKey` is already a uniform-random 32-byte output of Argon2id, so no extra HKDF salt is needed.
- **Symmetric envelope crate — XChaCha20-Poly1305 (slice (c), shipped 2026-04-29).** Deliberate deviation from `RAPPORT-STATE-MODEL.md` §6.6's `age` recommendation. `age`'s primary modes (X25519 keypairs or scrypt-from-passphrase) do not naturally accept a 32-byte HKDF-derived domain key without either re-applying scrypt (wasteful, doubles KDF cost on every read) or deriving an indirection X25519 keypair (extra abstraction layer). The ChaCha20-Poly1305 family is the natural primitive for "AEAD given a 32-byte key" — same family used by Signal, WireGuard, TLS 1.3. The **X**ChaCha variant (24-byte nonce vs. plain ChaCha's 12) was chosen so random-per-call nonce generation is safe for the operator's lifetime — no per-key message counter to track and no ~2³² birthday-bound to budget. The 12-byte cost per bundle is noise. Encryption granularity (per-domain with operator-derived master key) and the §6.4 plaintext/ciphertext split remain exactly as the doctrine specifies; only the envelope crate changes. See "Documentary debt to retire" below.
- **Bundle layout and AAD binding (slice (c)).** Bundle = 4-byte magic `b"COOE"` | 1-byte bundle version `0x01` | 1-byte aead id `0x02` (XChaCha20-Poly1305) | 24-byte nonce | ciphertext-with-appended-16-byte-tag. The 6-byte header is bound as AAD on every encrypt and decrypt — kills downgrade-style attacks that swap leading bytes between formats, and establishes the AAD plumbing so §3 can add semantic AAD (row id, etc.) as a v2 bundle bump rather than a public-API change. Failure mode is uniform: framing checks (length, magic, version, aead id) yield `CryptoError::MalformedBundle` with a `&'static str` reason; everything else collapses to `CryptoError::Aead` (tag-mismatch and wrong-key not publicly distinguished). Bundle layout is locked by `kat_pinned_layout` in `crypto::envelope::tests` — any drift in layout, AEAD variant, AAD scheme, or upstream `chacha20poly1305` output breaks the KAT loudly.

### Resolved during Phase 1 §3 (2026-04-29)

- **Vault module placement (slice (a)).** Top-level `vault` module at `src-tauri/src/vault.rs`, sibling to `crypto/` and `db/`. Not under `crypto::*` — the vault is the first *consumer* of the crypto substrate, not another primitive. Module name `vault` chosen over `lock`/`init` because the eventual scope covers more than the lock-state machine (the operator's encrypted state lifecycle).
- **Lock-key derivation primitive (slice (a)).** `crypto::derive_lock_key(&MasterKey) -> DomainKey` over HKDF-SHA256 with info `coo/v1/lock`. Deliberately sibling to — not a member of — the four state-domain derivations: the `Domain` enum still enumerates exactly the four `RAPPORT-STATE-MODEL.md` §1 state domains, and the lock primitive is a parallel construct because the lock is not state. The `coo/v1/lock` info string sits in a deliberately distinct namespace from `coo/v1/kdf/<domain>`. Locked by `kat_lock` in `crypto::derive::tests` — renaming orphans every existing install's sentinel.
- **Sentinel design (slice (a)).** File at `<coo_dir>/sentinel` (sibling to `<coo_dir>/salt`), bytes are exactly the AEAD bundle produced by `crypto::envelope::encrypt(derive_lock_key(master), SENTINEL_PLAINTEXT)`. No outer wrapping — the bundle's own framing from §2 (c) is already self-describing. 62 bytes total for the 16-byte sentinel plaintext. Locked by `vault::tests::sentinel_kat_pinned_layout`, which simultaneously locks the lock-key info string, the sentinel plaintext, and the envelope output for a fixed master + fixed nonce.
- **Sentinel plaintext (slice (a)).** `b"COO-LOCK-SENT-01"` (16 ASCII bytes). The "-01" suffix is the rotation hedge — a future v0.x sentinel could change the constant deliberately without ambiguity. Renaming after data lands in production orphans every existing install's sentinel.
- **Sentinel filename (slice (a)).** Plain `sentinel` in the operator's `~/.coo/` directory, alongside `salt`. Both follow the same plaintext-sibling-file precedent set by `~/.coo/salt` in §2 (a) — the file's bytes are already self-protecting (the bundle's AEAD handles authentication; the salt file's framing handles structural integrity).
- **Failure-mode translation at the unlock boundary (slice (a)).** `crypto::envelope::decrypt`'s uniform `CryptoError::Aead` collapse is preserved at the envelope layer. `vault::unlock` translates `Aead → UnlockError::WrongPassphrase` *at this single boundary only*, because that is the user-facing meaning of "the sentinel did not authenticate." A tampered sentinel file therefore reads to the user as `WrongPassphrase`; we cannot distinguish "wrong key" from "tag mismatch" without storing more, and storing more breaks the passphrase-not-stored rule (`RAPPORT-STATE-MODEL.md` §6.2). Accepted. `CryptoError::MalformedBundle` translates to `UnlockError::Inconsistent` so the caller can distinguish "wrong passphrase" from "corrupted on-disk state."
- **`InitState` shape (slice (a)).** `Uninitialized` (neither salt nor sentinel present) | `Initialized` (both present) | `Inconsistent(&'static str)` (exactly one present). Exactly-one cases report `"missing sentinel"` or `"missing salt"` so the caller can render a meaningful error rather than silently re-initializing. `setup_passphrase` refuses to run from anything other than `Uninitialized`; `unlock` refuses to run from anything other than `Initialized`.
- **Slice cut for §3 (b) (single-shot, 2026-04-29).** §3 (b) shipped as one migration with three tables rather than splitting into b1 (plaintext config + encrypted-column pattern proven on `operator_profile`) and b2 (`calibration_setting`). The encrypted-column pattern is exercised twice in the same diff — once on a singleton table (`operator_profile`), once on a key/value table (`calibration_setting`) — which proves the convention is reusable without paying the overhead of two commits. Both end-to-end tests live in `db::tests` and exercise the §3 (a) vault substrate end-to-end (`setup_passphrase` → `domain_key(OperatorKnowledge)` → `encrypt` → INSERT → SELECT → `decrypt` → assert).
- **Calibration typing deferred to §6 (slice (b)).** `EXILE.md` §3 enumerates ten calibration dials cleanly into four groups (Texture: Warmth / Directness / Mystery; Posture: Persistence / Argument / Conviction / Aggressive; Currency: Flirtation; Foundation: Devotion / Discipline). What §3 does *not* lock is the dials' internal representation: endpoints are named pairs (`cool ↔ open`), but §4's "Familiar" preset uses hand-tuned per-dial labels (`Warmth: present`, `Mystery: held`, `Aggressive: protective`, `Flirtation: low — present`) that don't share a uniform shape. Picking enum-vs-float-vs-step in §3 (b) and re-typing in §6 (Calibration surface) would be a non-additive migration (`ALTER TABLE ... ALTER COLUMN ... SET DEFAULT` is not supported in SQLite anyway); deferring to §6 is the §7-friendly answer. The §3.5 doctrinal ceiling at calibration 4b is **not** a dial position — it's a meta-constraint on `warmth_register` transitions in the state model and an inference-assembly clamp (§5.2 step 5), neither of which §3 (b) touches. The placeholder `calibration_setting` table is `(dial_key TEXT PK, ciphertext BLOB, schema_version, updated_at)` — `dial_key` is plaintext (the dial names are public doctrine — `EXILE.md` §3.1–§3.4), the chosen value is the encrypted payload.
- **Encrypted-column convention (slice (b)).** Plaintext identity columns + a single `ciphertext BLOB NOT NULL` column carrying the full §2 (c) v1 AEAD bundle (6-byte header + 24-byte nonce + ct+tag). Both encrypted tables in §3 (b) (`operator_profile`, `calibration_setting`) use `Domain::OperatorKnowledge` for the domain key — calibration is operator-state-shaped and doesn't fit `Rapport`/`FriendshipFloor`/`Conversation` per `RAPPORT-STATE-MODEL.md` §1. Future state-domain tables (rapport state, friendship floor, conversation history, operator-knowledge appendable entries) follow the same convention but key under their respective domain. `updated_at` is plaintext for ordering / audit — leaks "operator changed something at time T" but no content.
- **Singleton-via-CHECK for `operator_profile` (slice (b)).** `id INTEGER PRIMARY KEY CHECK (id = 1)` turns ADR-0011's single-operator commitment into a SQL-layer invariant rather than relying on application code to never write a second row. Locked by `db::tests::operator_profile_singleton_enforced` — second-row INSERT must be rejected by SQLite.
- **`schema_version` is required-explicit (slice (b)).** `schema_version INTEGER NOT NULL` with **no DEFAULT**. `RAPPORT-STATE-MODEL.md` §7.2's versioned-envelope discipline turns on every row committing to its own version; defaulting to 1 hides that commitment and would let a sloppy v2 caller silently write v1 rows after a non-additive migration. Friction is the feature.
- **No semantic AAD on §3 (b) bundles.** The §2 (c) AAD is just the 6-byte header. Adding row-identity AAD (e.g., binding ciphertext to `(table_name, row_pk)` so an attacker can't swap a bundle from row A into row B's BLOB column) is a v2 bundle bump that requires extending the public envelope API to take an `aad: &[u8]` parameter, a v2 bundle id, KAT churn, and a v1→v2 lazy-read migration. None of that fits §3 (b)'s schema-only scope. Tracked as documentary debt for follow-up.

## Documentary debt to retire

- **`RAPPORT-STATE-MODEL.md` §6.6 envelope crate recommendation.** §6.6 names `age` for envelope encryption. The §2 (c) implementation uses **XChaCha20-Poly1305** instead — `age`'s primary modes do not naturally fit a 32-byte HKDF-derived domain key, and the XChaCha variant's 24-byte nonce removes the random-nonce safety bound that plain ChaCha20-Poly1305 carries. The bundle layout (4-byte magic + 1-byte version + 1-byte aead id + 24-byte nonce + ciphertext+tag) and the header-as-AAD binding are also additions made in §2 (c) that §6.6 does not specify. Encryption granularity (per-domain), `MasterKey` derivation (Argon2id), domain-key derivation (HKDF), and the §6.4 plaintext/ciphertext split are unchanged from the doctrine. Reasoning is preserved in "Resolved during Phase 1 §2" above. Land in a v0.x minor of `RAPPORT-STATE-MODEL.md` when convenient — natural moment is Phase 1 close (§2 already closed without it).
- **In-memory hygiene of decrypted plaintext.** `crypto::envelope::decrypt` returns plaintext as a plain `Vec<u8>`. `MasterKey` and `DomainKey` are `Zeroizing`-wrapped, but the cleartext leaves the crypto layer in an unwrapped allocation. `RAPPORT-STATE-MODEL.md` §6.3 covers what's encrypted at rest and §6.5 covers the wire surface, but neither speaks to in-memory hygiene of decrypted values. §3 callers will inherit this constraint silently. Worth folding into the same `RAPPORT-STATE-MODEL.md` v0.x minor that retires the §6.6 debt above.
- **Lock-key primitive and unlock UX-layer error translation (slice §3 (a)).** §3 (a) introduces two extensions that `RAPPORT-STATE-MODEL.md` §6 does not contemplate. (1) A passphrase-verification *lock key* derived as a sibling to the four §1 state-domain keys — HKDF info `coo/v1/lock`, used to encrypt a fixed sentinel plaintext at `<coo_dir>/sentinel`. §6 only describes per-state-domain encryption; the lock primitive lives parallel to those four domains and is not itself state. (2) An unlock-boundary error translation that maps `CryptoError::Aead → UnlockError::WrongPassphrase` and `CryptoError::MalformedBundle → UnlockError::Inconsistent` *at the `vault::unlock` boundary only* (the envelope layer keeps its uniform `Aead` collapse). §6 does not address UX-layer error mapping. Neither extension is a deviation from doctrine — both are extensions to what §6 underspecified. The technical record is preserved in "Resolved during Phase 1 §3" above. Land in the same `RAPPORT-STATE-MODEL.md` v0.x minor that retires the §6.6 envelope-crate debt and the in-memory-hygiene debt — natural moment is Phase 1 close.
- **v2 bundle bump for semantic AAD (slice §3 (b)).** §2 (c) ships v1 bundles with the 6-byte header bound as AAD. §3 (b) writes those v1 bundles into `operator_profile.ciphertext` and `calibration_setting.ciphertext` without further AAD. An attacker with write access to the SQLite file can swap a valid bundle from one row into another row's BLOB column — the AEAD will accept it (same key, same domain) because nothing binds ciphertext to row identity. The threat model is integrity-against-on-disk-tampering (defense in depth on top of the OS-level file protections); not a primary concern, but the §2 (c) bundle layout was designed to absorb semantic AAD as a v2 bundle bump rather than a public-API change. Implementing this requires: (1) extending the public envelope API to take an `aad: &[u8]` parameter; (2) bumping the bundle id from `0x01` to `0x02` (or rolling a new aead id) with a corresponding KAT bump; (3) a v1→v2 lazy-read migration so existing rows continue to decrypt under v1 while new writes use v2; (4) updating the §3 (b) tables' INSERT call sites to compute `aad = (table_name, row_pk)` or similar deterministic identity. Out of scope for §3 (b) (schema-only). Natural retire moment is when the next state-domain table lands (Phase 1 §6) — design the v2 bundle once, apply uniformly across all encrypted-column tables.
