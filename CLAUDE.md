## This Product's Stack

- **Tauri** for the desktop application shell (Rust backend, web frontend)
- **Rust** for the backend half (encryption, SQLite access, inference abstraction client) — the operator is taking on Rust as part of this project; the ramp is bounded per `../doctrine/decisions/0011-coo-as-independent-product.md` Consequences section
- **Frontend framework** — TBD at Phase 1 entry (React / Svelte / Solid candidates; not gated by doctrine)
- **API inference via Anthropic Claude** at MVP, behind an `InferenceProvider` abstraction layer enabling provider swap (and eventually local inference) without application rewrite
- **Local SQLite** under `~/.coo/`
- **Encrypted at rest** — per-domain encryption with operator-derived master key; `age`/`rage` recommended; `argon2id` for KDF
- **No Docker.** COO is a native desktop app per the non-negotiable in `mvp/coo.md`. The simplicity is the feature.

## Ports

COO is a native desktop application; it does not expose ports for external consumption. Internal Tauri IPC between the Rust backend and the web frontend uses Tauri's native command channel — no TCP ports.

If a future feature requires a local listening port (e.g., for development tooling, debugging, or a future dobackbone-connection surface that needs callback URLs), that port must be allocated from the band reserved for COO in `../doctrine/docker/PORTS.md` via an ADR. **As of Phase 1 MVP, no ports are required.**

## Product-Specific Rules

These are load-bearing for COO and apply only inside this repository.

- **`EXILE.md` Sections 1 and 1.5 are permanent.** They do not change after finalization. If a session generates content or proposals that would alter the character text, the session has drifted. Surface and stop.
- **The wife-protection rule is structural.** Per `EXILE.md` §2.5 and `RAPPORT-STATE-MODEL.md` §2.5, the operator's wife is not in the application as content, as a field, as a category, or as a topic Exile originates. The schema does not provide a slot that invites her in. The operator may bring her up; Exile listens. Exile never initiates.
- **The doctrinal ceiling at calibration 4b is enforced in two layers** per `RAPPORT-STATE-MODEL.md` §3.5. The state model never reaches values that would translate to past-4b shaping; the inference assembly pipeline includes an explicit ceiling-clamp step. Defense in depth. A future session that proposes raising the ceiling is making a doctrine change, not a tuning change.
- **No silent writes by the AI.** Any change Exile proposes to the operator's data — Dossier knowledge entries, Briefs, Kit inventory — requires explicit operator confirmation. Per `RAPPORT-STATE-MODEL.md` §2.3 and §3.3.
- **Encryption at rest is non-negotiable.** Rapport state, friendship floor, operator-knowledge, and conversation history are encrypted at rest. The operator-derived passphrase is not stored; if the operator forgets it, state is unrecoverable. This is a real and accepted cost.
- **State stays in `~/.coo/`.** No primary persistence outside this directory. No external services required for offline operation.
- **No Docker.** Native desktop app; the simplicity is the feature.
- **Model version pinning.** The model behind Exile does not change silently. COO pins to specific model versions; upgrades are deliberate releases tested against the doctrine, not automatic events.
- **In-flight context is a named surface, not a hidden one.** When inference happens via API, Exile's context — character text, rapport state summarized, conversation history — crosses the wire. This surface is acknowledged. Provider choice is governed by data-handling guarantees. Prompt construction commits to minimization (only context necessary for the current turn crosses).
- **The personal-commander tier sits above the dobackbone-officer tier when connected.** Per ADR-0011 and `mvp/coo.md`. Exile speaks for the operator to dobackbone's officers. The officers do not change because Exile is there. (Phase 2 work; not in MVP scope.)

### Resolved during Phase 1 §2 (2026-04-29)

- **Argon2id parameters** — m=65536 (64 MiB), t=3, p=1. RFC 9106's "second recommended option" with parallelism reduced to 1 for portability; targets `RAPPORT-STATE-MODEL.md` §6.2's 250–500ms band on desktop hardware. Params are persisted into the salt file alongside the salt itself, so a future tuning change does not orphan existing installs (old salts re-derive their old keys).
- **Salt file format** — 34-byte framed structure at `~/.coo/salt`: magic `COO1` + file version + kdf id + little-endian m/t/p costs + 16-byte salt. Plaintext per `RAPPORT-STATE-MODEL.md` §6.4. Magic, version, and kdf id are all checked on load; malformed files are rejected before any KDF work.
- **`MasterKey` API** — `pub struct MasterKey(Zeroizing<[u8; 32]>)`. Only access path is `expose_secret() -> &[u8; 32]`. No `Debug`, no `Clone`, no public constructor; test sites use a `#[cfg(test)] pub(crate) MasterKey::from_bytes_for_test` so KATs do not pay the Argon2id cost on every test run. Production callers must go through `derive_master_key`.
- **HKDF info-string namespace** — `coo/v1/kdf/<domain>` for each of the four `RAPPORT-STATE-MODEL.md` §1 domains (`rapport`, `friendship-floor`, `operator-knowledge`, `conversation`). The `v1` segment is the rotation hedge: bumping to `v2` is a controlled re-derivation event under the operator's passphrase, not a silent code change. Renaming any info string after data lands in production orphans every existing ciphertext for that domain — KATs in `crypto::derive` lock the strings against accidental change.
- **HKDF salt** — `None`, per RFC 5869 §3.1. The `MasterKey` is already a uniform-random 32-byte output of Argon2id, so no extra HKDF salt is needed.
- **Symmetric envelope crate — XChaCha20-Poly1305 (slice (c), shipped 2026-04-29).** Deliberate deviation from `RAPPORT-STATE-MODEL.md` §6.6's `age` recommendation. `age`'s primary modes (X25519 keypairs or scrypt-from-passphrase) do not naturally accept a 32-byte HKDF-derived domain key without either re-applying scrypt (wasteful, doubles KDF cost on every read) or deriving an indirection X25519 keypair (extra abstraction layer). The ChaCha20-Poly1305 family is the natural primitive for "AEAD given a 32-byte key" — same family used by Signal, WireGuard, TLS 1.3. The **X**ChaCha variant (24-byte nonce vs. plain ChaCha's 12) was chosen so random-per-call nonce generation is safe for the operator's lifetime — no per-key message counter to track and no ~2³² birthday-bound to budget. The 12-byte cost per bundle is noise. Encryption granularity (per-domain with operator-derived master key) and the §6.4 plaintext/ciphertext split remain exactly as the doctrine specifies; only the envelope crate changes. See "Documentary debt to retire" below.
- **Bundle layout and AAD binding (slice (c)).** Bundle = 4-byte magic `b"COOE"` | 1-byte bundle version `0x01` | 1-byte aead id `0x02` (XChaCha20-Poly1305) | 24-byte nonce | ciphertext-with-appended-16-byte-tag. The 6-byte header is bound as AAD on every encrypt and decrypt — kills downgrade-style attacks that swap leading bytes between formats, and establishes the AAD plumbing so §3 can add semantic AAD (row id, etc.) as a v2 bundle bump rather than a public-API change. Failure mode is uniform: framing checks (length, magic, version, aead id) yield `CryptoError::MalformedBundle` with a `&'static str` reason; everything else collapses to `CryptoError::Aead` (tag-mismatch and wrong-key not publicly distinguished). Bundle layout is locked by `kat_pinned_layout` in `crypto::envelope::tests` — any drift in layout, AEAD variant, AAD scheme, or upstream `chacha20poly1305` output breaks the KAT loudly.

### Resolved during Phase 1 §3 (2026-04-29)

- **Vault module placement (slice (a)).** Top-level `vault` module at `src-tauri/src/vault.rs`, sibling to `crypto/` and `db/`. Not under `crypto::*` — the vault is the first *consumer* of the crypto substrate, not another primitive. Module name `vault` chosen over `lock`/`init` because the eventual scope covers more than the lock-state machine (the operator's encrypted state lifecycle).
- **Lock-key derivation primitive (slice (a)).** `crypto::derive_lock_key(&MasterKey) -> DomainKey` over HKDF-SHA256 with info `coo/v1/lock`. Deliberately sibling to — not a member of — the four state-domain derivations: the `Domain` enum still enumerates exactly the four `RAPPORT-STATE-MODEL.md` §1 state domains, and the lock primitive is a parallel construct because the lock is not state. The `coo/v1/lock` info string sits in a deliberately distinct namespace from `coo/v1/kdf/<domain>`. Locked by `kat_lock` in `crypto::derive::tests` — renaming orphans every existing install's sentinel.
- **Sentinel design (slice (a)).** File at `<coo_dir>/sentinel` (sibling to `<coo_dir>/salt`), bytes are exactly the AEAD bundle produced by `crypto::envelope::encrypt(derive_lock_key(master), SENTINEL_PLAINTEXT)`. No outer wrapping — the bundle's own framing from §2 (c) is already self-describing. 62 bytes total for the 16-byte sentinel plaintext. Locked by `vault::tests::sentinel_kat_pinned_layout`, which simultaneously locks the lock-key info string, the sentinel plaintext, and the envelope output for a fixed master + fixed nonce.
- **Sentinel plaintext (slice (a)).** `b"COO-LOCK-SENT-01"` (16 ASCII bytes). The "-01" suffix is the rotation hedge — a future v0.x sentinel could change the constant deliberately without ambiguity. Renaming after data lands in production orphans every existing install's sentinel.
- **Sentinel filename (slice (a)).** Plain `sentinel` in the operator's `~/.coo/` directory, alongside `salt`. Both follow the same plaintext-sibling-file precedent set by `~/.coo/salt` in §2 (a) — the file's bytes are already self-protecting (the bundle's AEAD handles authentication; the salt file's framing handles structural integrity).
- **Failure-mode translation at the unlock boundary (slice (a)).** `crypto::envelope::decrypt`'s uniform `CryptoError::Aead` collapse is preserved at the envelope layer. `vault::unlock` translates `Aead → UnlockError::WrongPassphrase` *at this single boundary only*, because that is the user-facing meaning of "the sentinel did not authenticate." A tampered sentinel file therefore reads to the user as `WrongPassphrase`; we cannot distinguish "wrong key" from "tag mismatch" without storing more, and storing more breaks the passphrase-not-stored rule (`RAPPORT-STATE-MODEL.md` §6.2). Accepted. `CryptoError::MalformedBundle` translates to `UnlockError::Inconsistent` so the caller can distinguish "wrong passphrase" from "corrupted on-disk state."
- **`InitState` shape (slice (a)).** `Uninitialized` (neither salt nor sentinel present) | `Initialized` (both present) | `Inconsistent(&'static str)` (exactly one present). Exactly-one cases report `"missing sentinel"` or `"missing salt"` so the caller can render a meaningful error rather than silently re-initializing. `setup_passphrase` refuses to run from anything other than `Uninitialized`; `unlock` refuses to run from anything other than `Initialized`.
- **Slice cut for §3 (b) (single-shot, 2026-04-29).** §3 (b) shipped as one migration with three tables rather than splitting into b1 (plaintext config + encrypted-column pattern proven on `operator_profile`) and b2 (`calibration_setting`). The encrypted-column pattern is exercised twice in the same diff — once on a singleton table (`operator_profile`), once on a key/value table (`calibration_setting`) — which proves the convention is reusable without paying the overhead of two commits. Both end-to-end tests live in `db::tests` and exercise the §3 (a) vault substrate end-to-end (`setup_passphrase` → `domain_key(OperatorKnowledge)` → `encrypt` → INSERT → SELECT → `decrypt` → assert).
- **Calibration typing deferred to §6 (slice (b)).** `EXILE.md` §3 enumerates ten calibration dials cleanly into four groups (Texture: Warmth / Directness / Mystery; Posture: Persistence / Argument / Conviction / Aggressive; Currency: Flirtation; Foundation: Devotion / Discipline). What §3 does *not* lock is the dials' internal representation: endpoints are named pairs (`cool ↔ open`), but §4's "Familiar" preset uses hand-tuned per-dial labels (`Warmth: present`, `Mystery: held`, `Aggressive: protective`, `Flirtation: low — present`) that don't share a uniform shape. Picking enum-vs-float-vs-step in §3 (b) and re-typing in §6 (Calibration surface) would be a non-additive migration (`ALTER TABLE ... ALTER COLUMN ... SET DEFAULT` is not supported in SQLite anyway); deferring to §6 is the §7-friendly answer. The §3.5 doctrinal ceiling at calibration 4b is **not** a dial position — it's a meta-constraint on `warmth_register` transitions in the state model and an inference-assembly clamp (§5.2 step 5), neither of which §3 (b) touches. The placeholder `calibration_setting` table is `(dial_key TEXT PK, ciphertext BLOB, schema_version, updated_at)` — `dial_key` is plaintext (the dial names are public doctrine — `EXILE.md` §3.1–§3.4), the chosen value is the encrypted payload.
- **Encrypted-column convention (slice (b)).** Plaintext identity columns + a single `ciphertext BLOB NOT NULL` column carrying the full §2 (c) v1 AEAD bundle (6-byte header + 24-byte nonce + ct+tag). Both encrypted tables in §3 (b) (`operator_profile`, `calibration_setting`) use `Domain::OperatorKnowledge` for the domain key — calibration is operator-state-shaped and doesn't fit `Rapport`/`FriendshipFloor`/`Conversation` per `RAPPORT-STATE-MODEL.md` §1. Future state-domain tables (rapport state, friendship floor, conversation history, operator-knowledge appendable entries) follow the same convention but key under their respective domain. `updated_at` is plaintext for ordering / audit — leaks "operator changed something at time T" but no content.
- **Singleton-via-CHECK for `operator_profile` (slice (b)).** `id INTEGER PRIMARY KEY CHECK (id = 1)` turns ADR-0011's single-operator commitment into a SQL-layer invariant rather than relying on application code to never write a second row. Locked by `db::tests::operator_profile_singleton_enforced` — second-row INSERT must be rejected by SQLite.
- **`schema_version` is required-explicit (slice (b)).** `schema_version INTEGER NOT NULL` with **no DEFAULT**. `RAPPORT-STATE-MODEL.md` §7.2's versioned-envelope discipline turns on every row committing to its own version; defaulting to 1 hides that commitment and would let a sloppy v2 caller silently write v1 rows after a non-additive migration. Friction is the feature.
- **No semantic AAD on §3 (b) bundles.** The §2 (c) AAD is just the 6-byte header. Adding row-identity AAD (e.g., binding ciphertext to `(table_name, row_pk)` so an attacker can't swap a bundle from row A into row B's BLOB column) is a v2 bundle bump that requires extending the public envelope API to take an `aad: &[u8]` parameter, a v2 bundle id, KAT churn, and a v1→v2 lazy-read migration. None of that fits §3 (b)'s schema-only scope. Tracked as documentary debt for follow-up.
- **Slice cut for §3 (c) (c1 Rust IPC + c2 React UI, 2026-04-29).** §3 (c) split into two sub-slices because the layers are different domains and review surfaces. (c1) is pure Rust: managed `AppState`, six `#[tauri::command]` wrappers, three pure-Rust `db::put_*` write helpers separated from the wrappers so they remain unit-testable without a Tauri runtime. (c2) is pure TypeScript/React: top-level routing in `App.tsx`, multi-step wizard, unlock screen, error surfaces. (c1) is the load-bearing IPC surface review (commands are hard to change without breaking the frontend); (c2) is UX iteration. Single-shot would have been defensible but the split lets each commit stand alone.
- **`AppState` shape (slice (c1)).** `pub struct AppState { coo_dir: PathBuf, db: Mutex<Connection>, vault: Mutex<Option<UnlockedVault>> }`. Resolved once at startup via `vault::default_coo_dir` + `db::open_and_migrate`, registered via `app.manage(...)` in the Tauri setup hook. The `MasterKey` never crosses the IPC wire; it lives in `vault` for the lifetime of the unlocked session. `Mutex<Option<UnlockedVault>>` is `Send + Sync` because `UnlockedVault: Send` (its `MasterKey` is a `Zeroizing<[u8; 32]>`-backed newtype with no non-`Send` fields).
- **Lock ordering convention (slice (c1)).** Vault before db. The two are never held simultaneously: write commands extract the per-domain key from the vault (lock vault, derive key, drop vault lock) and *then* take the db lock. `DomainKey` is owned (HKDF-SHA256 produces fresh bytes per call, `Zeroizing`-wrapped), so it crosses the lock boundary safely. Eliminates deadlock potential between commands without forcing a uniform fat-lock strategy.
- **`InspectResult` shape (slice (c1)).** JSON-tagged enum: `{ "status": "uninitialized" }` | `{ "status": "initialized", "onboarding_completed": bool }` | `{ "status": "inconsistent", "reason": "..." }`. Bundles `vault::InitState` with the `app_config[onboarding_completed_at]` existence check so JS makes one round-trip to decide route between onboarding / unlock / main, not two. The `onboarding_completed_at` value itself (the ISO timestamp) is not surfaced — only its presence; the field is operator-state of "have they completed onboarding?", not "when?", per the post-(d) routing concern.
- **Wizard step structure (slice (c2)).** Four steps: welcome → passphrase (with ≥ 8 char minimum + confirmation match) → profile (callsign defaulted to `Cardinal-7` per `EXILE.md` §9.5; optional display name) → review. Final commit issues writes in order: `theme=secret_agent` (plaintext), `operator_profile` (encrypted JSON of `{callsign, display_name}`), 10 `calibration_setting` rows (Familiar preset from `EXILE.md` §4 voice-sample header — `warmth=present`, `mystery=held`, `discipline=exacting`, etc.), `onboarding_completed_at=<iso>` (the §3 (d) startup-gating marker). No per-dial UI per the calibration-typing-deferred-to-§6 call from slice (b).
- **Mid-onboarding crash recovery deferred to §3 (d) (slice (c2)).** If the operator sets passphrase + writes theme + crashes before profile/calibration commit, on relaunch `vault_inspect` returns `Initialized + !onboarding_completed`. (c2) treats this as the standard unlock path (UnlockScreen → ChannelPlaceholder); the wizard does not auto-resume from where it crashed. Real recovery (re-inspect after unlock; route to wizard at the profile step if `!onboarding_completed`) is §3 (d) startup-gating work.
- **Wizard-resume on mid-crash (slice (d)).** App.tsx now routes on the (status, onboarding_completed, unlocked) triple instead of just (status, unlocked). The new branch is `initialized + unlocked + !completed → OnboardingWizard(startStep="profile")` — operator unlocks the existing vault on UnlockScreen, then the wizard mounts at the profile step (skipping welcome + passphrase, which would fail since vault_setup refuses to run on an already-initialized vault). The wizard's `commit()` is unchanged: every write is an UPSERT per the §3 (c1) `put_*` helpers, so any partial pre-crash writes are idempotently overwritten. App.tsx's wizard `onComplete` now refreshes `vault_inspect` instead of just toggling local state — first-run path also benefits, since post-wizard state is read from disk rather than inferred. No Rust changes; the §3 (c1) IPC surface already exposes everything the resume path needs.
- **§3 closes (slice (d)).** All four sub-slices shipped 2026-04-29: (a) vault module + lock-key + sentinel; (b) schema migration #2; (c1) Tauri IPC surface + managed AppState; (c2) React wizard + unlock screen; (d) startup gating + wizard-resume. Phase 1 §3 is closed; §4 (Channel surface) is the next entry point.

### Resolved during Phase 1 §5 (2026-04-30)

- **Sequencing — §5 first, then §4 (2026-04-30).** §4 (Channel surface) needs an inference provider to actually talk to Claude; §5's `InferenceProvider` is load-bearing per ADR-0011 ("API inference behind an abstraction layer enabling provider swap"). Getting the trait shape right before any UI calls into it is cheap at design time and expensive once §4/§6 surfaces have built on top. Reverse order — §4 with a hardcoded stub inline, §5 carving the abstraction out later — was considered and rejected: risks the trait shape being warped by whatever the channel surface happened to need first.
- **Default model + model selection (slice (a)).** Default `claude-opus-4-7`, overridable via env var `COO_INFERENCE_MODEL` at startup. The trait stays model-agnostic; the model id is the Claude impl's concern (read at construction in §5 (b)). Opus over Sonnet/Haiku for the operator's MVP because Exile's character-holding and calibration-aware response shaping per `EXILE.md` §1 / §1.5 + the §4.A intimacy-register ladder is where smaller-model drift would show first; the cost premium is acceptable for a single-operator personal app. Operator can downgrade later by exporting `COO_INFERENCE_MODEL=claude-sonnet-4-6` (or whatever the future answer is) without rebuilding.
- **API key handling (slice (a)).** Plaintext env var `ANTHROPIC_API_KEY`. Considered but rejected: encrypting the key under a sibling lock-key (HKDF info `coo/v1/api-key`) at `<coo_dir>/api-key`. Encrypted-key story would have followed the §3 (a) sentinel precedent and kept the key off plaintext disk, but it adds a key-rotation surface and an unlock-time decrypt step that doesn't earn its complexity for a single-operator personal app. Env var keeps key management in the operator's hands (shell profile, .env file outside the repo, secret manager — operator's choice). When `ANTHROPIC_API_KEY` is unset, `build_provider()` falls back to the stub provider so the app still runs for UI-only work.
- **Trait shape (slice (a)).** `InferenceProvider: Send + Sync` with one `async fn infer(&self, InferenceRequest) -> Result<InferenceResponse, InferenceError>` method. Non-streaming-only at §5 (a); streaming (`infer_stream`) is an additive future addition — a default-implemented method on the same trait, overridden by the Claude impl when §4 needs incremental rendering. Adding a default-implemented method to a trait does not break existing implementors. `async_trait` macro for trait-object dispatch (native async fn in traits doesn't yet allow `Box<dyn Trait>` for async methods); the `Pin<Box<Future>>` indirection is irrelevant for network-bound inference. Locked by `inference::tests::build_provider_returns_a_working_trait_object`.
- **Request/response shape (slice (a)).** `InferenceRequest { system_prompt: String, messages: Vec<Message> }` — system prompt held separate from messages because Claude's API treats it that way and because `RAPPORT-STATE-MODEL.md` §5.2's inference assembly pipeline produces it as a distinct artifact (EXILE.md §1 + §1.5 + §2 verbatim, state-derived prose modifiers, calibration ceiling clamp, wellbeing posture). Putting it in `messages` would mean every consumer reconstructs the split, inviting bugs. `Message { role: Role, content: String }` with `Role` an enum of `User | Assistant` matching Claude's two-role conversation model. `InferenceResponse { content: String }` — minimal at §5 (a); usage tokens / model id / stop reason are additive when callers need them.
- **Error model (slice (a)).** `InferenceError` with four variants: `Provider(String)` (model overloaded, content filter, anything we can't more specifically classify); `Network(String)` (connection failure, timeout); `Auth(String)` (401/403 — most likely missing or invalid `ANTHROPIC_API_KEY`); `RateLimited` (429). Lets the §4 channel surface render distinct messages: "try again" vs. "check your API key" vs. "wait a moment". Stub never errors; §5 (b) Claude impl maps anthropic-side errors into these.
- **Stub provider behavior (slice (a)).** Echoes the last user message back with `"[stub] you said: "` prefix; falls back to `"[stub] you said: (no user message)"` when no user message is present. Slightly more useful than a fixed canned string — §4 development can verify the system prompt and message history are reaching the provider correctly. Tests pin all three paths (last-user-wins, no-messages, assistant-only).
- **`build_provider()` and AppState wiring (slice (a)).** Single constructor returning `Box<dyn InferenceProvider>`. §5 (a) always returns the stub; §5 (b) extends to read `ANTHROPIC_API_KEY` and dispatch to a Claude impl when set. `AppState.inference` is `Box<dyn InferenceProvider>` (no `Mutex` — the trait is Send + Sync and `&dyn InferenceProvider` is shared safely across Tauri command threads; concurrent `infer` calls don't conflict at the abstraction layer). Populated in `lib.rs::run` setup hook alongside `coo_dir` / `db` / `vault`.
- **HTTP client and mock server (slice (b)).** `reqwest` 0.12 with `default-features = false, features = ["rustls-tls", "json"]` for the production path; `wiremock` 0.6 as the dev-dep that backs §5 (b)'s tests. `reqwest` chosen over `ureq` because the trait's `async fn infer` is the natural shape for a network-bound seam (and Tauri 2 already runs a tokio executor) — `ureq`'s sync API would force a `spawn_blocking` shim inside `infer` to satisfy the signature. `rustls-tls` over `native-tls` keeps the build OpenSSL-free, which matters for the Windows Tauri target. `wiremock` over `mockito` because it is async-native and composes with the existing `#[tokio::test]` setup; `mockito`'s sync API would force more ceremony for the request-shape assertions.

Community-maintained crates (e.g. `anthropic-sdk`) exist but introduce a transitive maintenance dep on a non-Anthropic-owned package for a multi-year-horizon personal app. The Messages API surface §5 (b) needs is small — one POST against `/v1/messages`, four request fields (`model`, `max_tokens`, `system`, `messages`), one response shape (`content` as a list of typed blocks), four error-status branches. Hand-rolled `#[derive(Serialize)]` / `#[derive(Deserialize)]` types living next to the impl in `inference/claude.rs` is ~40 lines of straightforward code with full control over the surface. Re-evaluate if Anthropic ships an official SDK or if §6 needs surfaces (streaming, tool use, prompt caching) that change the cost calculus.
- **`ClaudeProvider` shape (slice (b)).** `pub struct ClaudeProvider { client: reqwest::Client, api_key: String, model: String, base_url: String }`. `reqwest::Client` cached on the struct rather than built per call — the client owns the connection pool and reuses TLS sessions, both of which would be wasted on per-call construction. `base_url` defaulting to `"https://api.anthropic.com"` but injectable via a `#[cfg(test)]`-only `new_with_base_url` constructor so the wiremock-backed tests can point the client at a local ephemeral port without ever hitting Anthropic. Public `new(api_key, model)` is the only production-callable constructor; `base_url` is otherwise inaccessible.
- **Wire schema and outbound headers (slice (b)).** Outbound body: `{ "model": <model>, "max_tokens": 4096, "system": <system_prompt>, "messages": [{"role": "user"|"assistant", "content": <text>}] }`. `max_tokens` defaults to **4096** — Anthropic requires the field; 4096 is sized for prose at the register `EXILE.md` §1 / §1.5 establishes (measured, not maximalist) and is well within Opus 4.7's per-turn ceiling. Promoting `max_tokens` to a per-request override is an additive future change once §4 / §6 surface a need. Response parsing pulls `content[]` and concatenates every block whose `type == "text"`, ignoring non-text blocks (`tool_use`, etc. — out of scope until §6 wires tool use). Headers: `x-api-key: <key>`, `anthropic-version: 2023-06-01`, `content-type: application/json`. The version pin (`2023-06-01`) is the Anthropic-published API version string, not the model id; locked here so a future Anthropic API revision doesn't silently change request semantics. Renaming the version string is a deliberate doctrine-touching change.
- **Error mapping (slice (b)).** `401` and `403` both map to `InferenceError::Auth(message)` — most §5 (a) failures will be missing or invalid `ANTHROPIC_API_KEY`, and the §4 channel surface should render both identically ("check your API key"). `429` maps to `InferenceError::RateLimited` (no message — the variant carries no data, matching the §5 (a) trait shape). All other non-success statuses (`5xx`, other `4xx`) map to `InferenceError::Provider(message)` — model overloaded, content filter, malformed-from-our-side request, anything else. Transport failures (connection refused, DNS, TLS) map to `InferenceError::Network(message)` via `reqwest::Error::to_string()`. Anthropic's structured error envelope (`{"type": "error", "error": {"type": "...", "message": "..."}}`) is parsed when present so the operator-facing message reaches the channel; falls back to `"HTTP <status>: <body>"` when the envelope is missing or malformed (e.g. infrastructure-layer 502s from upstream proxies).
- **`build_provider_from_env` testability seam (slice (b)).** Public `build_provider()` reads `ANTHROPIC_API_KEY` and `COO_INFERENCE_MODEL` from process env once and hands off to a private `build_provider_from_env(api_key: Option<String>, model_override: Option<String>) -> Box<dyn InferenceProvider>`. The split keeps env-reads at one well-defined point and lets the dispatch logic be unit-tested deterministically without mutating process env (which would force `serial_test` for parallel-safe tests). Empty-string treatment: `Some("")` for either env var is treated identically to `None` — operator can `unset ANTHROPIC_API_KEY` or `export ANTHROPIC_API_KEY=` without the empty string reaching `ClaudeProvider::new` as a literal API key, and similarly for the model id.
- **Slice cut for §5 (b) (single-shot, 2026-04-30).** §5 (b) shipped as one commit rather than splitting into b1 (request serialization + happy path) / b2 (error mapping) / b3 (build_provider dispatch + env vars). The components are tightly coupled: request serialization isn't end-to-end testable without the success-path mock; error mapping needs a built provider to error from; `build_provider_from_env` is six lines of `match` over `Option`. The whole diff lands at ~330 lines of impl + tests in `inference/claude.rs` plus ~30 lines of dispatch + tests in `inference.rs`. Compare to §3 (c) which split because c1 (Rust IPC) and c2 (React UI) were genuinely different review surfaces; §5 (b) is one Rust module, one trait impl, one constructor extension, all the same domain.

### Resolved during Phase 1 §4 (2026-05-01) — slice (b)

- **In-window cap (`INFERENCE_WINDOW_TURNS = 100`) ships with the substrate.** §4 (b) implements tier 1 of `RAPPORT-STATE-MODEL.md` §4.1's three-tier retention model. Tiers 2 (within-session summaries) and 3 (cross-session summaries) ship in §4 (c). The cap is `db::INFERENCE_WINDOW_TURNS` as a named constant — single source of truth, tunable. Operator-visible: turns past the window are not sent to inference but remain on disk and visible in the UI. Doctrinally aligned with the retention model rather than substrate-only-ship-then-fix-later. The §4 (c) job is replacing the dropped turns with in-character summaries — same seam, focused job.
- **Two `list_turns_*` functions rather than parameterized.** `list_turns_for_ui` returns all turns; `list_turns_for_inference` returns the most recent `INFERENCE_WINDOW_TURNS`. Callers don't make the choice at call sites — the UI load path always wants everything, the inference path always wants the window. Naming the paths clearly is cheaper than naming a parameter clearly.
- **`infer` command IPC contract changed.** §4 (a1) shipped `infer(messages: Vec<Message>) -> String`; §4 (b) ships `infer(session_id, operator_turn) -> InferResponse` with the message history read from disk inside the command. Disk becomes the source of truth; React state is a projection. The §4 (a1) shape had one consumer (`ChannelSurface.dispatch`) and re-shaping it now is cheap; re-shaping after §6 surfaces had built on it would not be. Locked by new `commands::tests::infer_response_wire_shape_is_pinned`. The §4 (a1) `inference_command_error_wire_shape_is_pinned` KAT continues to pass — `InferenceCommandError` is unchanged, with db/crypto failures inside `infer` collapsing into `Provider("conversation persistence: ...")` via a `From<DbError>` impl. The collapse is lossy at the surface (the React side cannot distinguish a transient network error from a db write failure) but accepted at §4 (b) scope: on-disk failure modes are vanishingly rare in this slice's writes (encrypted-write of a UTF-8 string under a derived domain key) and the operator-actionable response is the same — try again or surface to the implementer if it persists.
- **No `conversation_summary` table in migration #3.** `RAPPORT-STATE-MODEL.md` §2.4 specifies the summary table as part of the retention model. §4 (b) does not exercise it — summaries are not written until §4 (c) implements summarization. Migrating the table now would pre-commit schema shape that hasn't been tested against real conversation flow. §4 (c) will add migration #4 with the summary table when it's actually needed. Same restraint as §3 (b) deferring calibration typing to §6.
- **`role` is plaintext, content is encrypted.** `conversation_turn.role` is a CHECK-constrained TEXT column rather than encrypted. The role is structural (it shapes the inference request), not sensitive. Decrypting on every read just to know which speaker would cost real and protect nothing. Same logic as §3 (b)'s `app_config.key` and `calibration_setting.dial_key` being plaintext. `created_at` is also plaintext for ordering and date-divider rendering — same trade.
- **Lock ordering preserved across the rewired `infer` command.** Vault before db, never simultaneous. The `infer` command takes vault lock briefly to derive the domain key, drops it, takes db lock + transaction to write the operator turn and read the in-window history, commits and drops the db lock, makes the network call without holding any locks, then re-takes vault and db locks (fresh domain-key derivation as defense in depth — cheap HKDF-SHA256 against the master key already in memory) to write the assistant turn. Same convention as §3 (c1); applies cleanly to the longer flow.
- **Transactional writes for turn + session count.** `infer` and `append_turn` wrap the `put_turn` INSERT and the `conversation_session.turn_count` UPDATE in a single SQLite transaction. Atomicity matters here — a half-write would leave `turn_count` out of sync with the actual rows. `next_turn_index` is computed inside the same transaction (`SELECT COALESCE(MAX(turn_index), -1) + 1`) so two concurrent writers can't allocate the same index; the UNIQUE (session_id, turn_index) constraint catches any future regression of that property loudly.
- **Server-side timestamps via SQLite `strftime`, not a `chrono` / `time` dep.** All conversation timestamps come from `db::current_iso_timestamp` which runs `SELECT strftime('%Y-%m-%dT%H:%M:%fZ', 'now')` — UTC, ISO 8601 with millisecond precision and explicit `Z` suffix. JS `new Date(...)` round-trips it; React's date-divider helper parses it locally. Considered and rejected: adding `chrono` (heavy transitive deps) or rolling a manual ISO formatter against `std::time::SystemTime` (timezone math). Letting SQLite be the canonical clock keeps the dep graph small and gives one source of truth across all writers (including future surfaces).
- **`uuid` v4 dep added for session + turn ids.** Per `RAPPORT-STATE-MODEL.md` §2.3 / §2.4 precedent (`operator_knowledge_entry.id` is a UUID; conversation tables follow). `uuid = { version = "1", features = ["v4"] }`. UUIDs let future surfaces reference turns / sessions without coordinating an integer-id namespace, and v4 random generation is cheap and collision-safe at any scale this app will see.
- **Optimistic operator-turn rendering with retry-detection.** The React side appends the operator's turn to local state with `turn_index = -1` before the IPC round-trip completes, so the bubble appears at typing latency. On `infer` success, the optimistic turn is replaced with the disk-authoritative turn carrying the real `turn_index`. On error, the optimistic turn is preserved (operator doesn't retype) and the existing retry UI handles the recovery. The retry path detects "last turn is optimistic with this same content" and skips appending another optimistic turn — keeps local state clean across error → retry → success cycles. Slice plan-acknowledged limitation: if the operator-turn write succeeded but the inference call failed, the operator turn IS on disk; retry calls `infer` again which writes *another* operator turn. The React state stays normalized; the rare on-disk dup is acceptable at §4 (b) scope. A no-dup retry path would require either a separate `retry_infer(session_id)` command (out of slice plan scope) or content-matching at the Rust side (fragile).
- **Test count overage by +2.** Plan estimated ~82 (3 schema + 5 helper + 2 wire-shape = 10 new); shipped 84 (added `next_turn_index_starts_at_zero_and_advances` and `latest_session_id_returns_most_recent_by_started_at`). Both extras lock useful behavior in public helpers consumed by the commands; without them, regressions in those helpers would only fail through integration paths. The "~" prefix in the slice plan's test counts allows the slack.
- **`append_turn` shipped despite §4 (b) not consuming it.** The slice plan §3.3 specifies the command exists. `ChannelSurface.dispatch` writes turns through `infer` rather than calling `append_turn` directly, so at §4 (b) the only test coverage for `append_turn` is its wire-shape KAT (`AppendTurnResponse` is included by virtue of the command surface) and the integration flow uses the same db helpers underneath. Surfacing the command keeps the IPC contract complete for future surfaces (e.g. a manual test harness or a Phase 2 dobackbone-driven entry point that needs to write a turn without going through the inference provider).

### Resolved during Phase 1 §4 (2026-05-01) — slice (c)

- **Summarization trigger — hybrid threshold-batch + session-end.** Within-session summarization fires when unsummarized turns exceed `INFERENCE_WINDOW_TURNS + SUMMARIZATION_BATCH_SIZE` (100 + 30 = 130). The oldest 30 turns are summarized in one call. At session end (inactivity gap closes a session per the boundary-detection helper), all remaining unsummarized turns are rolled into a cross-session summary as part of the session-close path. Hybrid avoids per-turn-check latency hits on every turn while ensuring no session ever ends with raw turns lying around uncompressed.
- **Session boundary — inactivity gap.** Sessions roll on inactivity rather than per-app-launch or per-calendar-day. `SESSION_INACTIVITY_GAP_HOURS = 6` is the threshold. Captures conversational rhythm without timezone math: evening to next morning is a new session, lunch break to afternoon is the same session. Single named constant; tunable based on observed behavior.
- **Cross-session boundary detection in `infer` only — IPC contract reading.** Slice plan §2.1 / §3.1 says `infer` detects cross-session boundaries on incoming turn; §3.5 says the §4 (b) IPC contract is unchanged. The two can't both be strictly true: handling boundary inside `infer` requires the response to carry the new session_id (or React holds a stale id). Resolved by reading "IPC contract unchanged" as "no new commands, signatures stable" — and treating extending `InferResponse` with a `session_id` field as an additive wire-shape change. The §4 (b) `infer_response_wire_shape_is_pinned` KAT is updated. `load_conversation` does NOT detect cross-session boundaries; the path that catches "operator returns after 6+ hours" is: relaunch → load_conversation → unlock → ChannelSurface mount → user sends a turn → `infer` detects the gap on the first call. The `app stays open 6+ hours` edge case lands cleanly because the next `infer` call sees the gap and rolls. After `infer` returns a session_id different from the local one, the React side reloads conversation state from disk so the new session view + the freshly-written cross-session summary stanza land in scrollback.
- **Summary content presented as synthetic assistant message, not system-prompt injection.** Per `RAPPORT-STATE-MODEL.md` §4.2, summaries are *Exile's own recollection*, not external context. The inference assembly prepends summaries as a single synthetic assistant message before the in-window turns rather than mixing them into the system prompt or wrapping them around the operator's message. This preserves the doctrinal layering: system prompt = identity + surface discipline, conversation context = what's been said + what's been remembered. Locked because changing it later would re-shape every inference call.
- **Summarization is in-character.** The summarization call uses `EXILE.md` §1+§1.5+§2 + the §4 (a3) output discipline + a new summarization directive. Exile summarizes herself in her own register. This is the durability commitment from `RAPPORT-STATE-MODEL.md` §4.2 in runtime form: across years, what survives is *her* recollection, not a generic compression layer's output.
- **The summarization directive is a separate constant from `OUTPUT_DISCIPLINE`.** `prompt::SUMMARIZATION_DIRECTIVE` instructs Exile that this is recall, not live speech. The directive explicitly notes that the output discipline (no third-person, no stage directions) governs *her live speech*, not her recollection — a summary may legitimately use third-person about the operator without violating discipline. This nuance is intentional and pinned by `prompt::tests::summarization_prompt_contains_directive_pin`.
- **No new Tauri commands.** Summarization machinery runs inside the existing `infer` flow. The IPC surface remains the same six §3 (c1) commands plus `load_conversation` / `append_turn` / `infer` from §4 (b). React-side change is purely the `LoadConversationResponse` extension (`summaries` field), the `InferResponse` extension (`session_id` field), and rendering of `<SummaryStanza>` components. The operator can see exactly where summarization happened, in Exile's words, by scrolling back.
- **Synchronous summarization, no background workers.** When summarization is triggered, it runs synchronously inside `infer` before the operator-facing response. The latency hit is real but bounded — a one-time additional inference round-trip on every ~30th turn (within-session) or on the first turn after a 6+ hour gap (cross-session). Adding asynchrony (background queue, scheduled job, deferred summarization) would be meaningful complexity for a small UX win at MVP scale. Re-evaluate if observed latency is bad enough to warrant the complexity.
- **Five-phase `infer_impl` with bounded lock acquisition.** `infer` refactored into a thin `#[tauri::command]` wrapper around `infer_impl(db_mutex, vault_mutex, provider, session_id, operator_turn)`. Five phases: (1) locked — boundary detection, operator-turn write, threshold check, capture pending summarization batches; (2) no-locks — run summarization inference call(s); (3) locked — write summary rows + read in-window turns + summaries; (4) no-locks — operator-facing inference call; (5) locked — write assistant turn. Lock ordering (vault before db, never simultaneous) preserved across the longer flow; network calls hold no locks. The split also enables the integration KATs in `commands::tests` to drive the same flow without a Tauri runtime.
- **`db::test_support` module to expose migrations to `commands::tests`.** The production migrations function is private so prod callers go through `db::open_and_migrate`, which resolves the operator's home directory. Integration tests need an in-memory connection migrated to the latest schema without touching `~/.coo/`. New `#[cfg(test)] pub(crate) mod test_support` exposes `run_migrations(&mut Connection)` which the new `infer_command_*` tests use to set up the test fixture. Surfaced minimally: only the function the tests actually need.
- **Hand-rolled ISO-8601 second-precision parser, no `chrono` / `time` dep.** `db::detect_session_boundary` needs to compare two ISO-8601 UTC timestamps as differences in hours. The existing `db::current_iso_timestamp` produces them via SQLite's `strftime`; a 30-line hand-rolled parser using Howard Hinnant's civil-from-days algorithm converts to seconds-since-epoch. Adds no transitive deps. A future Phase 2 surface that needs richer time arithmetic (timezones, calendar reckoning, durations) can re-evaluate.
- **Cross-session summary's `generated_at` is the prior session's `ended_at`.** When the boundary fires, the summary is conceptually generated *as* the session ended, even though the inference call to produce it runs after. Using `ended_at` as `generated_at` keeps the chronological ordering in `list_summaries_for_inference` correct (cross-session summaries appear chronologically before any within-session summaries of subsequent sessions) and matches the React rendering which sorts cross-session summaries by `generated_at`.
- **Test count overage by +6.** Plan estimated ~93 (4 + 6 + 3 + 2 = 15 new); shipped 99. The slice-plan's "~93" appears to be an arithmetic underestimate against its own §2.4 + §3.6 itemized counts (the four §2.4 schema KATs + six db helper KATs + three prompt KATs + two commands KATs sum to 15 new, and 84 + 15 = 99). The "~" prefix in the slice plan's test counts allows the slack.

## Documentary debt to retire

- **v2 bundle bump for semantic AAD (slice §3 (b)).** §2 (c) ships v1 bundles with the 6-byte header bound as AAD. §3 (b) writes those v1 bundles into `operator_profile.ciphertext` and `calibration_setting.ciphertext` without further AAD. An attacker with write access to the SQLite file can swap a valid bundle from one row into another row's BLOB column — the AEAD will accept it (same key, same domain) because nothing binds ciphertext to row identity. The threat model is integrity-against-on-disk-tampering (defense in depth on top of the OS-level file protections); not a primary concern, but the §2 (c) bundle layout was designed to absorb semantic AAD as a v2 bundle bump rather than a public-API change. Implementing this requires: (1) extending the public envelope API to take an `aad: &[u8]` parameter; (2) bumping the bundle id from `0x01` to `0x02` (or rolling a new aead id) with a corresponding KAT bump; (3) a v1→v2 lazy-read migration so existing rows continue to decrypt under v1 while new writes use v2; (4) updating the §3 (b) tables' INSERT call sites to compute `aad = (table_name, row_pk)` or similar deterministic identity. Out of scope for §3 (b) (schema-only). Natural retire moment is when the next state-domain table lands (Phase 1 §6) — design the v2 bundle once, apply uniformly across all encrypted-column tables.
